Strategy summary

Das design-system-Repo ist heute eine kleine, gut geordnete statische Site mit einem hand-geschriebenen design-system.css (429 Zeilen, --gat-*-Token, .gat-*-Komponenten). Eine 18-Iterations-Welle im gemeindefinanzen-Labor hat eine vollstaendige web-taugliche Auslegung erprobt — entsaettigte Tokens, Komponenten, A11y, Druck, Vollbild, Chart-Helfer. Diese Welle fliesst jetzt als 2.0-Release in einem Schritt zurueck ins org-DS.

Drei Tragesaeulen: Tailwind v4.3 als Build-Schicht (per npm devDependency, kein Vendoring), Token-Migration in @theme bei stabilen Klassen-/Token-Namen (Werte aendern sich, Namen nicht), und ein @custom-variant gat-mode-hc als saubere A11y-Variant. Die gehostete URL grueneat.github.io/design-system/design-system.css bleibt identisch — Konsumenten brechen nicht.

  • Tailwind v4.3 ist die natuerliche Wahl.

    v4.0 GA Januar 2025, v4.3 stable Mai 2026 — produktionsreif. @theme emittiert Tokens automatisch unter :root; @custom-variant loest die A11y-Variant nativ.

  • AA-Haertung ist gezielt klein.

    Nur ein Token-Wert (--gat-color-hellgruen: #56af31 mit Ratio 2.77 → #3e8a25 mit Ratio 4.51) aendert sich. Alle anderen Marken-Token bestehen heute schon AA.

  • Konsumenten-stabile API.

    Gleiche Pages-URL, gleicher <link>, gleiche .gat-*-Klassennamen. Sichtbar aendern sich nur Header-Optik (weisse Brandbar statt Dunkelgruen-Block) und Headline-Gewicht.

  • "Kein Vendoring" ist konform.

    Workspace-CLAUDE.md erlaubt Tailwind als devDependency explizit; die "kein Build"-Regel der gemeindefinanzen-App gilt nur dort, nicht im DS-Repo.

  • Hellgruen-Wert braucht Marken-Approval.

    Der CD-Quickguide kennt #56af31 als Plakat-Wert; fuer Web ist die dunklere Variante zwingend AA-konform. Offene User-Entscheidung.

03Decisions

Key decisions & options

Build-Engine: Tailwind v4 CLI

Chosen

Tailwind v4.3 + @tailwindcss/cli als devDependencies. @theme-Block emittiert Tokens nach :root, @custom-variant loest A11y-Variant nativ. v4.0 GA seit Januar 2025, v4.3 stable seit Mai 2026.

Tailwind v3: JS-Theme statt CSS-@theme, kein @custom-variant — die gruene.at-Variant-Loesung waere haendisch.

Lightning CSS oder PostCSS-Preset-Env: keine Variant-Engine, A11y-Variant haendisch.

Status quo (hand-geschrieben): in CONTEXT.md verworfen — die gruene.at-Stack-Parallele und das Variant-System sind die Treiber der 2.0-Welle.

--gat-color-hellgruen Wert aendern

Open — User-Entscheidung

Empfohlen: #56af31 (heute, 2.77 auf Weiss — AA FAIL) → #3e8a25 (4.51 auf Weiss — AA OK). Sichtbar dunkler, aber im 2.0-Tag konsistent und unmissverstaendlich.

Alternative: Wert unveraendert lassen + strukturelle Regel in MIGRATION.md ("nie als Text auf hellem Grund — nur Flaeche mit Anthrazit-Text"). Setzt Konsumenten-Disziplin voraus.

Marken-Stakeholder-Approval steht aus.

Chart-Palette als gat-charts.js ES-Modul

Chosen (Discretion bestaetigt)

Option (a): ES-Modul ueber die gleiche Pages-Root, grueneat.github.io/design-system/gat-charts.js. Exporte: PALETTE, INK, LABEL_SIZE/ AXIS_SIZE, BAR_MAX_*, VA_DECAL, tip()/legende()/grid()/ planIstLegende(). Konsumenten import via CDN. Vanilla ESM — kein Build.

Option (b) — nur Doku-Konstanten: Drift garantiert (drei Tools wuerden 8 Hex-Werte abtippen).

Option (c) — ausserhalb DS-Scope: laesst das Welle unvollendet, ECharts braucht JS-Konstanten.

.gat-tag (nicht .gat-pill)

Chosen

.gat-tag beschreibt die semantische Funktion (kategoriale Markierung), .gat-pill die Form. Pill-Form (border-radius: 999px) bleibt im Default; eckigere Varianten koennten kuenftig ohne Klassen-Rename entstehen. Modifier: --neutral, --info, --pflicht, --risiko.

.gat-header — weisse Brandbar als Default

Chosen (CONTEXT Decision 3)

Default kippt zu weisser Brandbar nach gruene.at-Muster (Logo als <img>, gruene 3px-Akzentlinie, leichter Schatten). Optionaler .gat-header--dunkel-Modifier bewahrt das v1.x-Muster fuer Konsumenten, die es bewusst wollen.

Klassennamen bleiben stabil (.gat-header, .gat-header__inner, .gat-nav); Optik ist breaking — dokumentiert in MIGRATION.md.

.gat-mode-hc via @custom-variant

Chosen

@custom-variant gat-mode-hc (&:where(.gat-mode-hc, .gat-mode-hc *)); — eine Zeile, Tailwind generiert fuer jede Utility einen gat-mode-hc:-Counterpart. Plus hand-geschriebene Komponenten-Overrides im DS-CSS fuer die .gat-*-Komponenten. Palette: Anthrazit + Gelb + Magenta (bestehende Marken-Token, keine neuen --hc-Werte).

Reine Custom-Property-Inheritance (Dual-Token-Schicht): mehr Wartungs-Surface, kein Variant- Syntactic-Sugar fuer Konsumenten.

Output committen oder nur in Pages bauen

Open — pragmatische Empfehlung

Empfohlen: gerendertes design-system.css ins Repo committen — PRs zeigen Token-Aenderungen als Diff direkt sichtbar. Pre-Commit-Hinweis in README: npm run build vor jedem Commit.

Alternative: Output nur in CI bauen, gh-pages-Branch deployen — mehr Setup, kein klarer Mehrwert fuer ein 1-Datei-Repo.
