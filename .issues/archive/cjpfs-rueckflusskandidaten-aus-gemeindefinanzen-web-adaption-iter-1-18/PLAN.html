Strategy summary

Ein Tailwind-v4-getriebenes v2.0.0-Release des Grüne-AT Design Systems, das die im gemeindefinanzen-Labor (18 Iterationen) erprobte web-taugliche Auslegung als additive --gat-web-*-Schicht in das DS zieht und die heute fehlenden A11y- und Datenwerkzeug-Komponenten ergänzt. Der Pfad: erst das Werkzeug, dann die Inhalte, dann die Doku, dann der Tag.

  • Build-Pipeline zuerst, dann Migration.

    Task 1 stellt Tailwind v4 + CI-Drift-Guard auf; Task 2 zieht alle bestehenden --gat-*-Token pixel-identisch in den @theme-Block. Keine sichtbare Änderung bis Task 3.

  • Additive Token-Schicht statt Umbenennung.

    Bestehende --gat-*-Namen bleiben stabil (Aliase im @layer base). Neue Werte landen unter --gat-web-* — Konsumenten wählen die Schicht.

  • Konsumenten-URL bleibt unverändert.

    grueneat.github.io/design-system/design-system.css liefert weiter das gerenderte CSS — nur jetzt aus dem CI-Build statt handgepflegt. Die einzige sichtbare Optik-Änderung: weißer Header (Brandbar) statt Dunkelgrün-Block.

  • Atomare Commits, jeder Task einzeln rückrollbar.

    13 inhaltliche Commits + 1 Tag. CI prüft pro Commit via git diff --exit-code design-system.css, dass Source und Output übereinstimmen — kein Drift zwischen src/ und Output.

  • HC-Toggle-Knopf bleibt Konsumenten-Sache.

    DS liefert nur die @custom-variant gat-mode-hc plus hand-geschriebene Komponenten-Overrides. Konsumenten bauen den Toggle-Button selbst (Beispiel-Snippet in MIGRATION.md).

03Decisions

Strategische Optionen — alle bereits durch User entschieden (CONTEXT.md Decisions 1–5).

Key decisions & options

Build-Engine

Chosen: Tailwind v4 + CLI

Tailwind v4 + @tailwindcss/cli: liefert @custom-variant nativ (für .gat-mode-hc), emittiert @theme-Tokens automatisch unter :root (Konsumenten-Aufruf stabil), spiegelt den gruene.at-Stack. CLI statt PostCSS-Plugin — kein zweiter Config-File.

Tailwind v3: kein @custom-variant, JS-Theme-Config statt CSS-@theme — falsche Richtung für ein Token-zentriertes DS.

Lightning CSS standalone: keine Variant-Engine — .gat-mode-hc müsste handgestrickt werden.

Status quo (handgeschriebenes CSS): verliert alle Tailwind-Benefits, die das 2.0-Programm motivieren.

Chart-Palette-Auslieferung

Chosen: gat-charts.js ES-Modul via Pages

ECharts liest kein CSS — die Palette muss als JS-Konstanten geliefert werden. Ein gehostetes ES-Modul auf der gleichen Pages-Root ist die einzige nicht-Drift-Lösung.

Nur Hex-Konstanten in der Doku, jede App kopiert: garantiert Drift, sobald drei Tools die 8 Werte abtippen.

npm-Package @grueneat/design-charts: Overkill ohne Publish-Infrastruktur.

Built CSS commit-Policy

Chosen: Output committen + CI-Drift-Guard

Output (design-system.css) wird ins Repo committed; CI führt npm run build && git diff --exit-code design-system.css auf jedem PR/Push. Konsumenten brauchen kein npm; PR-Reviewer sehen Token-Änderungen direkt im Diff. Decision 5.

Output nur in Pages bauen, nie committen: Reviewer müssten den Diff selbst aus dem CI-Log rekonstruieren — unhandlich für ein 1-Datei-Repo.

Hellgrün-AA-Strategie

Chosen: Wertänderung #56af31 → #3e8a25

Wert wandert (sichtbar dunkler, aber AA-konform: 4.51 statt 2.77 auf Weiß). Strukturelle Regel zusätzlich in MIGRATION.md: Hellgrün ist Akzent-Fläche, nicht Textfarbe auf hellem Grund. Decision 5.

Wert unverändert lassen + nur strukturelle Disziplin-Regel: lässt sich praktisch nicht durchsetzen; bricht das Versprechen, dass DS-Token-Paare inhärent AA-konform sind.

Tabs/Switcher-Timing

Chosen: Im 2.0-Scope (eigene Task)

Decision 5 hebt .gat-tab* und .gat-switch* explizit ins 2.0-Release. Eigene Task (9) — nicht an Task 7/8 angeflanscht. Folder-Tab-Optik, Lesbarkeits-Mindestgrößen (Tabs 1.08rem, Switcher 0.98rem).

Späterer Folge-Issue: User-Entscheidung sagt anders.

HC-Toggle-Knopf-Ownership

Chosen: Konsument baut den Toggle

DS-Side schmal: @custom-variant + Komponenten-Overrides. Konsumenten kennen ihren Header-Kontext besser (gemeindefinanzen hat z. B. einen eigenen Knopf in der Brandbar). MIGRATION.md zeigt das JS-Snippet. Decision 5.

DS liefert .gat-mode-hc-toggle-Knopf: koppelt die Komponente an einen Header/Brandbar-Slot, den nicht jeder Konsument hat.
