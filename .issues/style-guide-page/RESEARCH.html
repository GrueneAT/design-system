01Strategy

Eine HTML-Datei, kein JS, gelockte Decisions.

Strategy summary

Issue #5 baut die heutige Skelett-index.html zu einem vollständigen, sichtbaren Style Guide aus — eine einzige deutschsprachige HTML-Seite, ohne JavaScript, die das gehostete design-system.css selbst einbindet und dogfooded. Die gesamte Arbeitsfläche ist bereits bekannt: ein CSS-File, ein HTML-File, ein Logo-Asset.

  • Lückenlose Inventur ist das Kernkriterium.

    design-system.css enthält 22 --gat-*-Tokens und 22 .gat-*-Klassen. Der Guide muss jedes Token und jede Klasse abbilden.

  • Sechs Abschnitte, Anchor-TOC ohne JS.

    Farben, Typografie, Layout, Komponenten, CD-Layout-Prinzipien, Maschinenlesbare Spezifikation — navigiert per nativen <a href="#anker"> in der .gat-nav.

  • Echte Komponenten vs. Doc-Chrome sauber trennen.

    Komponenten live mit gat--Klassen gerendert (Dogfooding); Swatches/Code-Blöcke/Do-Don't als doc--Klassen im inline <style> (D-2).

  • Vorbild: das persönliche System.

    Zweiteilige Gliederung Grundlagen → Komponenten, live gerenderte Beispiele — aber der Grüne-AT-Guide ergänzt Anchor-TOC + JSON-Spec und hat eigene grüne Identität.

02Flow

Task flow & dependencies

Die sechs Bau-Schritte der index.html entsprechen exakt den sechs Abschnitten der fertigen Seite — jeder ein Anker-Ziel des TOC.

Step 1 Kopf + inline <style> + Anchor-TOC design-system.css einbinden, doc-Chrome (D-2), Nav als TOC.
Step 2 Abschnitt Farben 14 Farb-Tokens als Swatch-Kacheln mit HEX + Zweck.
Step 3 Abschnitt Typografie 3 Schriften, 6 Größen, 2 Leadings, Live-Specimen.
Step 4 Abschnitt Layout Container, Grid-Varianten, Abstands-Skala.
Step 5 Abschnitt Komponenten Header/Nav, Buttons, Karten, CD-Elemente — live + Markup.
Step 6 CD-Layout + Maschinenlesbare Spezifikation Schutzzone/Logo/Typo-auf-Grün, dann JSON-Spec (D-1).

03Decisions

Key decisions & options

D-1 — Maschinenlesbare Spezifikation

Gelockt

Eingebetteter <script type="application/json">-Block in index.html — eine Datei, kein Sync-Risiko.

Separates design-system.json — zweite Datei, Drift-Gefahr.

D-2 — Doc-spezifisches CSS

Gelockt

Doc-Chrome (Swatches, Code-Blöcke, Do/Don't-Raster) als kleiner inline <style>-Block; design-system.css bleibt sauber auf Tokens + echte Komponenten beschränkt.

Doc-CSS in design-system.css — verschmutzt das gehostete, fremd-eingebundene Stylesheet.

Navigation & Smooth-Scroll

Empfehlung

Native <a href="#anker"> plus CSS scroll-behavior: smooth und scroll-margin-top auf den Abschnitts-Ankern — kein JavaScript, CONTEXT-konform.

Doc-Klassen-Präfix

Offen

Empfehlung doc- statt gat- für Doku-Möbel, damit die Trennung echter Komponenten von Doc-Chrome sichtbar bleibt. Vom Planner zu bestätigen.

Offene Frage Konkrete Objekt-Form des JSON-Spec-Blocks: D-1 legt das Format fest, der Entwurf (tokens / components / rules) ist eine begründete Empfehlung — der Planner finalisiert das Schema.
